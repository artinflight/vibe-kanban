---
title: "Local Backup and Recovery"
description: "Create a real, restorable backup of a local Vibe Kanban install before risky work"
---

This runbook is for the **local desktop/local server install** of Vibe Kanban.

Use it **before**:

- rebuilding the desktop or local server binary
- changing auth mode
- changing workspace paths, repo links, or database schema
- disabling cloud features
- restoring from an older snapshot
- asking an agent to perform broad recovery work

<Warning>
Copying only `db.v2.sqlite` is **not** a full backup.

If the local install is still coupled to a shared API or Electric stream, the local SQLite database may not contain the full issue and board graph you see in the UI. In that state, a "DB backup" is only a partial backup.
</Warning>

## What a real backup must include

A real local backup must capture all of these at the same time:

1. **Live application state**
   - `~/.local/share/vibe-kanban/`
   - every `db.v2*.sqlite` file
   - `config.json`
   - `credentials.json` if present
   - signing keys and other local runtime files
2. **Service and launch configuration**
   - `~/.config/systemd/user/vibe-kanban.service`
   - `~/.config/systemd/user/vibe-kanban.service.d/`
   - wrapper scripts such as `~/.local/bin/vibe-kanban-serve`
   - the currently deployed binary such as `~/.local/bin/vibe-kanban-server-cleanfix`
3. **Source state**
   - the checked out repo
   - `git status`
   - `git diff`
   - the current branch and current commit SHA
4. **Runtime evidence**
   - `/api/info`
   - service status
   - recent service logs
5. **Remote-coupling proof**
   - whether `/api/info` reports a non-null `shared_api_base`
   - whether the local UI is still using remote or Electric-backed issue streams

If any of these are missing, the backup is incomplete.

## The most important preflight check

Before calling anything a full backup, check whether the local install is still partially remote-backed:

```bash
curl -s http://127.0.0.1:4311/api/info | jq '.data.shared_api_base, .data.login_status, .data.remote_auth_degraded'
```

### Safe result

The safest state for a local-only backup is:

- `shared_api_base` is `null`
- local board data is known to persist in local tables
- the UI is not timing out on Electric streams before falling back

### Unsafe result

If `shared_api_base` is non-null, treat the install as **cloud-coupled** until proven otherwise.

That means:

- the local DB may only contain partial board state
- restoring only the local DB may lose issue columns, links, tags, assignees, and other board metadata
- you must not claim you have a full recovery point yet

<Warning>
If `shared_api_base` is non-null, stop and either:

1. export the remote-backed project data separately, or
2. first convert the install to fully local-only and verify that board state persists locally, then take the backup.

Do not proceed with risky recovery work on a cloud-coupled install and call the result "fully backed up".
</Warning>

## Standard backup procedure

Use a timestamped directory under `/home/<user>/backups/`.

Example:

```bash
ts=$(date -u +%Y%m%dT%H%M%SZ)
root="$HOME/backups/vk-full-backup-$ts"
mkdir -p "$root"
```

### Step 1: Record current runtime state

```bash
curl -s http://127.0.0.1:4311/api/info | jq '.' > "$root/api-info.json"
systemctl --user status vibe-kanban.service --no-pager -l > "$root/systemctl-status.txt" 2>&1 || true
journalctl --user -u vibe-kanban.service -n 400 --no-pager > "$root/journal.txt" 2>&1 || true
ps -eo pid,ppid,etime,%cpu,%mem,rss,comm,args --sort=-rss > "$root/processes.txt"
```

### Step 2: Back up the live local state directory

```bash
mkdir -p "$root/live-state"
cp -a "$HOME/.local/share/vibe-kanban/." "$root/live-state/"
```

Verify the copy:

```bash
find "$root/live-state" -maxdepth 2 -type f | sort > "$root/live-state-files.txt"
```

### Step 3: Back up service definitions and launchers

```bash
mkdir -p "$root/systemd"
cp -a "$HOME/.config/systemd/user/vibe-kanban.service" "$root/systemd/" 2>/dev/null || true
cp -a "$HOME/.config/systemd/user/vibe-kanban.service.d" "$root/systemd/" 2>/dev/null || true

cp -a "$HOME/.local/bin/vibe-kanban-serve" "$root/" 2>/dev/null || true
cp -a "$HOME/.local/bin/vibe-kanban-server-cleanfix" "$root/" 2>/dev/null || true
```

If the service points to a different binary, copy that too.

### Step 4: Back up repo state before edits

From the repo root:

```bash
git rev-parse --show-toplevel > "$root/repo-root.txt"
git branch --show-current > "$root/branch.txt"
git rev-parse HEAD > "$root/head.txt"
git status --short > "$root/git-status.txt"
git diff > "$root/git.diff"
git diff --stat > "$root/git-diffstat.txt"
git ls-files -o --exclude-standard > "$root/untracked.txt"
```

If worktrees are in use:

```bash
git worktree list --porcelain > "$root/worktrees.txt"
```

### Step 5: Record environment that affects behavior

```bash
systemctl --user cat vibe-kanban.service > "$root/systemd-cat.txt" 2>&1 || true
env | sort > "$root/env.txt"
```

If you are intentionally using a filtered environment, record the exact launch command separately.

### Step 6: Freeze a copy of the current binary identity

```bash
sha256sum "$HOME/.local/bin/vibe-kanban-serve" > "$root/sha256.txt" 2>/dev/null || true
sha256sum "$HOME/.local/bin/vibe-kanban-server-cleanfix" >> "$root/sha256.txt" 2>/dev/null || true
```

## Backup verification checklist

Do not say "backup complete" until all of these are true:

- the timestamped directory exists
- `api-info.json` exists
- the copied `live-state/` contains the current `db.v2.sqlite`
- service files and launchers were copied
- repo branch, HEAD, status, and diff were captured
- a file listing of the copied live-state directory exists
- the backup location was written into the task log or handoff
- the operator explicitly noted whether the install was cloud-coupled at backup time

## What does **not** count as a full backup

These are common failures:

- copying only `db.v2.sqlite`
- copying only the repo
- copying only the binary
- saying "backup taken" without recording `/api/info`
- taking a backup while `shared_api_base` is still non-null and pretending it is a full local recovery point
- taking a backup after risky edits instead of before

## Restore procedure

<Warning>
Restore into a quiescent system. Stop heavy agents and avoid changing the repo while restoring.
</Warning>

### Step 1: Stop the service

```bash
systemctl --user stop vibe-kanban.service
```

### Step 2: Preserve the broken current state before overwriting it

```bash
ts=$(date -u +%Y%m%dT%H%M%SZ)
rescue="$HOME/backups/vk-pre-restore-$ts"
mkdir -p "$rescue"
cp -a "$HOME/.local/share/vibe-kanban/." "$rescue/" 2>/dev/null || true
cp -a "$HOME/.config/systemd/user/vibe-kanban.service" "$rescue/" 2>/dev/null || true
cp -a "$HOME/.config/systemd/user/vibe-kanban.service.d" "$rescue/" 2>/dev/null || true
cp -a "$HOME/.local/bin/vibe-kanban-serve" "$rescue/" 2>/dev/null || true
cp -a "$HOME/.local/bin/vibe-kanban-server-cleanfix" "$rescue/" 2>/dev/null || true
```

### Step 3: Restore files from the chosen backup

```bash
backup="$HOME/backups/vk-full-backup-<timestamp>"

cp -a "$backup/live-state/." "$HOME/.local/share/vibe-kanban/"
cp -a "$backup/systemd/vibe-kanban.service" "$HOME/.config/systemd/user/" 2>/dev/null || true
cp -a "$backup/systemd/vibe-kanban.service.d" "$HOME/.config/systemd/user/" 2>/dev/null || true
cp -a "$backup/vibe-kanban-serve" "$HOME/.local/bin/" 2>/dev/null || true
cp -a "$backup/vibe-kanban-server-cleanfix" "$HOME/.local/bin/" 2>/dev/null || true
```

### Step 4: Reload and restart

```bash
systemctl --user daemon-reload
systemctl --user start vibe-kanban.service
```

### Step 5: Verify immediately

```bash
curl -s http://127.0.0.1:4311/api/info | jq '.'
systemctl --user status vibe-kanban.service --no-pager -l
journalctl --user -u vibe-kanban.service -n 200 --no-pager
```

Then verify:

- the expected binary is running
- the expected workspace and project count is present
- the expected board data is visible
- the install is still local-only if that was the intended state

## Required restore drill

If you are building a process for future agents, do not stop at "we copied files".

Run a restore drill:

1. take a backup
2. restore it into a test state or after a deliberate restart
3. verify the same issues, statuses, workspace links, and project counts are still present
4. document what persisted locally and what did not

If you have never completed that drill, you do not have a trusted backup process yet.

## Operational policy for agents

Any agent working on the local VK install must follow this policy:

1. Back up first.
2. Record whether the install is still cloud-coupled.
3. Do not call the backup "full" unless local persistence is proven.
4. Before schema changes, binary replacement, auth changes, or repo-link changes, take a fresh timestamped backup.
5. After each risky change, record the exact backup path in the handoff.

## If the install is still cloud-coupled

If `/api/info` still reports a shared API base, add these extra steps:

1. Record the exact `shared_api_base` value.
2. Record whether the UI still emits Electric timeout or fallback errors.
3. Export or otherwise preserve the remote-backed project data before proceeding.
4. Do not claim recovery is possible from SQLite alone.

This is the specific failure mode that causes "we took backups" and "we still cannot restore the board" to both be true.

## Minimum handoff note

Every risky VK task should end with a note like this:

```text
Backup created: /home/<user>/backups/vk-full-backup-<timestamp>
Service state captured: yes
Repo state captured: yes
/api/info captured: yes
shared_api_base at backup time: <value or null>
Local-only persistence verified: <yes/no>
Restore drill completed: <yes/no>
```

If that note does not exist, future operators should assume the backup is incomplete.
